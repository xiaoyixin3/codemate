# CodeMate 项目模块说明

## 一、项目整体结构

CodeMate 技术社区后端系统采用 Maven 多模块结构，将接口模型、核心工具、业务服务、Web 入口和前端资源进行拆分，便于后续维护和功能扩展。

当前改造策略是：项目对外展示名称为 CodeMate，代码内部暂时保留原有 paicoding 模块名和包名，避免一次性修改 Maven 模块、Java 包路径、Spring 扫描路径、MyBatis 配置和数据库脚本导致项目无法启动。

项目整体结构如下：

```text
paicoding
├── paicoding-api
├── paicoding-core
├── paicoding-service
├── paicoding-web
├── paicoding-ui
└── codemate-docs
```

## 二、Maven 模块说明

### 1. paicoding-api

`paicoding-api` 主要存放系统对外暴露和内部流转使用的数据模型，包括 DTO、VO、请求参数、响应对象等。

该模块的作用是统一接口入参和出参结构，降低 Controller、Service 和前端页面之间的数据耦合。后续整理接口文档时，可以优先从该模块梳理核心请求对象和响应对象。

### 2. paicoding-core

`paicoding-core` 是系统公共能力模块，主要包含通用工具类、缓存封装、异常处理、权限校验、公共枚举、基础配置等内容。

该模块相当于后端系统的基础设施层，给业务模块提供可复用能力。例如统一返回、全局异常、Redis 工具、用户上下文、AOP 扩展等能力，通常会放在这一层或由这一层支撑。

### 3. paicoding-service

`paicoding-service` 是核心业务逻辑模块，承载文章、评论、用户、点赞、收藏、关注、通知、搜索、排行榜等业务能力。

在实际开发中，Controller 层一般只负责参数接收、权限校验入口和结果返回，真正的业务编排、数据库访问、缓存读写、消息生成和排行榜统计逻辑主要放在该模块中。

### 4. paicoding-web

`paicoding-web` 是 Web 启动和接口入口模块，包含启动类、Controller、拦截器、过滤器、配置类、初始化逻辑等。

项目启动类位于：

```text
paicoding-web/src/main/java/com/github/paicoding/forum/web/QuickForumApplication.java
```

该模块负责接收 HTTP 请求，并将请求转发到 service 层完成业务处理。数据库初始化、Web MVC 配置、登录拦截、环境配置等也主要在该模块中管理。

### 5. paicoding-ui

`paicoding-ui` 主要存放前端静态资源和页面模板，用于支撑技术社区页面展示。

当前项目定位先做纯后端简历项目，因此该模块暂时不是改造重点。后续可以只保留必要页面展示，重点把接口能力、数据库设计、缓存优化和业务流程讲清楚。

### 6. codemate-docs

`codemate-docs` 是本次改造新增的个人项目文档目录，用来沉淀 CodeMate 的项目介绍、模块说明、数据库设计、接口文档、简历描述和后续改造计划。

该目录与原项目自带 `docs` 目录分开，避免和开源项目原始文档混在一起，也方便后续提交和展示。

## 三、核心业务模块说明

### 1. 用户模块

用户模块负责注册、登录、用户资料、用户关系和登录态管理。

简历中可以重点描述为：基于 JWT + 拦截器实现用户登录态校验，通过用户上下文封装当前登录用户信息，为文章发布、评论、点赞、收藏等需要登录的接口提供权限基础。

### 2. 文章模块

文章模块负责文章发布、编辑、删除、查询、分类、标签和阅读统计等能力。

这是技术社区系统的核心模块，可以作为项目主线讲解。后续可以围绕文章详情查询做 Redis 热点缓存，围绕阅读数、点赞数、收藏数做文章热度统计。

### 3. 评论模块

评论模块负责文章评论、评论回复、评论列表查询等能力。

评论和回复是社区互动的核心场景。后续可以将评论行为和消息通知模块打通，在用户评论或回复文章时生成站内通知。

### 4. 点赞收藏模块

点赞收藏模块负责用户对文章或评论的互动行为记录。

该模块适合体现高频写入和状态查询的处理能力。后续可以使用 Redis 缓存用户是否点赞、是否收藏等状态，减少重复查询 MySQL。

### 5. 关注模块

关注模块负责用户之间的关注关系维护，包括关注、取消关注、粉丝列表和关注列表等。

该模块可以和消息通知结合，在用户被关注时生成通知记录，也可以在后续扩展为关注用户动态流。

### 6. 消息通知模块

消息通知模块负责站内通知生成和查询，例如评论、回复、点赞、收藏、关注等行为产生的通知。

后续增强方向是增加未读消息统计、批量已读、按通知类型筛选等能力，让该模块更像真实业务系统中的通知中心。

### 7. 搜索模块

搜索模块负责技术内容的关键词检索。

当前可以先作为已有能力整理进文档，后续如果继续增强，可以补充文章标题、摘要、标签等维度的搜索能力，并整理成接口说明。

### 8. 排行榜模块

排行榜模块负责热门文章、活跃用户等榜单统计。

后续纯后端增强重点可以放在 Redis ZSet 热门文章排行榜：根据阅读、点赞、评论、收藏等行为计算文章热度分值，将结果写入 Redis ZSet，并通过定时任务同步或刷新榜单。

## 四、后端请求流程

典型后端请求流程如下：

```text
浏览器 / API 客户端
        ↓
paicoding-web Controller
        ↓
登录拦截器 / 参数校验 / 权限校验
        ↓
paicoding-service 业务逻辑
        ↓
paicoding-core 公共能力
        ↓
MySQL / Redis / Liquibase 初始化数据
        ↓
统一响应对象返回前端
```

以发布文章为例，请求进入 Controller 后，系统先通过拦截器识别当前登录用户，再由文章 Service 校验文章参数、保存文章基础信息、处理分类标签关系，最后返回文章发布结果。

以文章详情查询为例，后续可以优化为：先查询 Redis 热点文章缓存，缓存命中时直接返回；缓存未命中时查询 MySQL，并将结果写入 Redis，降低热门文章反复访问带来的数据库压力。

## 五、面试讲解版本

面试中可以这样介绍项目结构：

> CodeMate 是我基于开源技术社区项目改造的 Spring Boot 后端系统。项目采用 Maven 多模块结构，`api` 模块负责统一接口模型，`core` 模块负责公共工具、缓存、异常和权限等基础能力，`service` 模块承载文章、评论、用户、通知、搜索和排行榜等核心业务，`web` 模块负责 Controller、拦截器、配置和启动入口。当前我没有急着修改底层包名和工程结构，而是先把项目对外包装为 CodeMate，并逐步补充文档、接口说明、缓存优化和排行榜等后端能力，保证项目始终可以正常启动和演示。

也可以进一步强调自己的工作：

1. 梳理并维护项目多模块结构，明确 Controller、Service、公共能力和接口模型的职责边界。
2. 基于 JWT + 拦截器实现用户登录态校验，为文章、评论、点赞、收藏等接口提供权限控制。
3. 使用 Liquibase 管理数据库初始化脚本，解决本地空库启动时初始化流程异常的问题。
4. 计划引入 Redis 热点文章缓存和 Redis ZSet 排行榜，提升高频查询接口性能。
5. 整理项目文档、数据库文档和接口文档，将开源项目改造成可讲解、可展示的个人后端项目。
