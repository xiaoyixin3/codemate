# 面试演示指南

## 项目定位

CodeMate 是在开源社区项目基础上持续二次开发的程序员技术社区与 AI Agent 平台。面试时应明确说明继承部分和个人改造部分，不把原项目能力全部描述为从零开发。

## 三分钟演示

1. 打开 <http://43.136.86.251/>，展示社区首页、文章详情和用户互动。
2. 展示 Agent 工作台：创建任务、查看执行步骤、证据和运行轨迹。
3. 展示项目模块：
   `codemate-web -> codemate-service -> codemate-core -> codemate-api`。
4. 展示 GitHub Actions：Maven 测试、Playwright E2E、Compose 校验和容器发布。
5. 说明腾讯云部署链路：Nginx、应用容器、MySQL、Redis、Liquibase 和数据卷。

## 重点讲解

### 为什么使用多模块

- 隔离接口模型、基础设施、业务逻辑、页面资源和入口层。
- 控制依赖方向，避免 Controller、数据库访问和通用工具混杂。
- 支持按模块测试和构建。

### 为什么保留旧包名和数据库名

产品名与构建产物已统一为 CodeMate，但直接修改 Java 包名和 `pai_coding` 数据库名会影响 Spring 扫描、MyBatis 映射、Liquibase、历史数据和服务器配置。因此采用兼容保留、独立迁移的策略。

### Agent 如何降低不可靠输出

- 站内知识检索提供可追溯证据。
- 工具调用负责执行确定性动作。
- 会话与 Agent Run 持久化，记录步骤和状态。
- 对超时、失败和模型不可用设计降级、重试与取消。
- 使用离线评估和可复现演示验证关键链路。

### 如何保证发布安全

- GitHub Actions 先执行测试和配置校验。
- 镜像使用提交 SHA 标签，避免只依赖 `latest`。
- 国内网络无法稳定拉取 GHCR 时，使用同一提交构建的 JAR 手动发布。
- MySQL、Redis 和图片使用持久卷，应用替换不删除数据。
- 发布后检查启动日志与 HTTP 状态，保留上一版本用于回滚。

## 可诚实说明的不足

- 当前公网仍使用 IP 和 HTTP，尚未完成域名与 HTTPS。
- 应用和 Redis 健康检查、自动回滚、监控告警仍需完善。
- Dependabot 仍有需要分批升级和回归验证的依赖告警。
- JAR 和镜像体积较大，后续可使用分层 JAR 或 Jib 优化。
- 单机部署适合个人演示，不是高可用集群。

## 常见追问

### Redis 故障怎么办

认证撤销和缓存能力会受影响。生产设计应明确哪些数据允许降级、哪些安全校验必须失败关闭，并增加 Redis 健康检查、告警与恢复策略。

### Liquibase 有什么作用

它把数据库结构变更纳入版本控制，应用启动时按 changeset 顺序执行，减少手工执行 SQL 导致的环境差异。

### 为什么不用 Kubernetes

当前是单机个人项目，Docker Compose 成本更低、可解释性更强。只有在需要多副本、滚动发布、服务发现和集群资源调度时才引入 Kubernetes。

### 下一步优先做什么

先修复 Critical/High 依赖告警，再完成 HTTPS、健康检查、自动回滚、定时备份和监控，而不是继续堆叠无法稳定运维的新功能。
